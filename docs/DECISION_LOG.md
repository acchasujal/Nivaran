# DECISION_LOG.md

| Date | Decision | Reason | Alternatives Rejected | Impact |
|---|---|---|---|---|
| 2026-06-22 | Core concept = accountability/evidence layer, not a reporting+routing tool | Resolution-routing is the default sophisticated answer most analytically strong competitors will also reach; accountability/evidence reframe is less crowded and breaks the citizen-as-only-user assumption | Multi-agent resolution routing agent (v1 rec) | Defines entire product category and demo narrative |
| 2026-06-22 | Removed fabricated "ward health score" / officer ranking / resolution-rate metrics | Cannot be defended under a single judge question ("where did this number come from?"); presenting invented numbers as real performance data is a structural credibility risk, not a cosmetic one | Fabricated ward score dashboard (v2/v3 implementation) | Removes the single highest-risk element from the demo; impact layer rebuilt as evidence-based (Agent 3) |
| 2026-06-22 | Kept RTI Draft Generator and renamed "Councilor Brief" → "Community Issue Summary," rather than cutting both | RTI/escalation artifacts are the most memorable, genuinely agentic output in the system; the actual flaw was the legal-authority framing and the invented supporting data, not the artifact itself — fixed via disclaimer + evidence-only inputs | Cutting RTI and councilor brief entirely | Preserves demo memorability while removing legal/credibility exposure |
| 2026-06-22 | Agent 5 performs a real external action (email send / PDF export) rather than ending at display | A read→analyze→score→display pipeline is analytics, not agentic AI, per the project's own Agentic Depth self-audit; judges scoring 20% on this criterion will probe for real autonomy | 5-agent analytics-only pipeline | Satisfies Agentic Depth criterion defensibly; adds SendGrid/PDF build risk, mitigated by validating the real send early (Day 5, not Day 7) |
| 2026-06-22 | SQLite instead of Firebase | Faster solo build, avoids new auth/SDK surface area under time pressure | Firebase Realtime DB | Slightly less "Google ecosystem" breadth on the DB layer; offset by deeper Gemini Vision/function-calling usage elsewhere |
| 2026-06-22 | No authentication layer in hackathon scope | Time constraint; public read/write acceptable for a 7-day MVP demo | Full auth system | Documented explicitly as a known limitation, not hidden; rate-limited by IP to reduce abuse risk |
| 2026-06-22 | Asynchronous background tasks for Agent 3 & 4 at threshold | Cuts user-facing submission latency; executed via FastAPI BackgroundTasks (no separate queue worker process required) | Full synchronous pipeline for all agents / Celery background worker | Prevents timeout on POST /issues while keeping infrastructure simple |
